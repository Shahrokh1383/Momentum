### Backend Logic

- **API Routing Architecture (Context-Based):**
  - `routes/api/auth.php`: Wrapped in `Route::prefix('api/auth')`. Contains login, register, forgot-password, OAuth callbacks, and CSRF token endpoint. (No auth middleware).
  - `routes/api/user.php`: Wrapped in `Route::prefix('api/user')->middleware(['auth:sanctum', 'verified'])`. Contains all user features (habits, categories, profile, etc.).
  - `routes/api/admin.php`: Wrapped in `Route::prefix('api/admin')->middleware(['auth:sanctum', 'role:admin'])`. Contains admin features.
  
- **SPA Authentication (Sanctum + HttpOnly Cookies):**
  - Configure `config/sanctum.php` for SPA stateful authentication with `SESSION_DRIVER=cookie`.
  - `AuthController` (located in `App\Http\Controllers\Auth`): `register()`, `login()`, `logout()`, `me()`.
  - Return user object with `is_premium` computed attribute on every auth response.
  - CSRF token endpoint `/api/auth/sanctum/csrf-cookie` consumed by frontend on app load.

- **OAuth2 Integration (Google, Apple, GitHub):**
  - `OAuthController`: `redirect($provider)` generates OAuth URL and returns it to frontend.
  - `callback($provider)` receives authorization code, exchanges for token via provider API.
  - Logic: search `users` by `provider` + `provider_id`. If found, issue Sanctum cookie. If not, create user with `password = null`, set `provider`/`provider_id`, then issue cookie.
  - Frontend handles OAuth popup → code → sends to backend `/api/auth/{provider}/callback`.

- **Email Verification (Simulated):**
  - On `register()`, generate secure random token, store in `sent_emails_log` with `type = email_verification`.
  - `EmailVerificationController::verify($token)`: finds token in log, sets `users.email_verified_at = now()`.
  - `verified` middleware applied to the `api/user` route group to block unverified users.

- **RBAC (Role-Based Access Control):**
  - Custom middleware `RoleMiddleware` checking `auth()->user()->role` against allowed roles.
  - Route groups: `user` routes (default inside `api/user.php`), `premium` routes (wrapped in `EnsurePremium` middleware inside `api/user.php`), `admin` routes (inside `api/admin.php` with `role:admin`).
  - `EnsurePremium` middleware: checks `auth()->user()->subscription->isActive()` and `plan !== 'free'`. If false, abort 403 with JSON payload `{ error: 'subscription_required', message: 'Premium subscription required.' }`.

- **Rate Limiting:**
  - Apply `Throttle:5,1` on `login`, `register` (inside `api/auth`).
  - Apply `Throttle:1,5` on `forgot-password` and `reset-password` (inside `api/auth`).
  - Apply global `Throttle:60,1` on all `/api/user/` routes.
  - Apply `Throttle:10,1` on file upload endpoints.

- **Strict Password Validation:**
  - In `AuthController::register()` and password reset logic, enforce strict backend rules: `Password::min(8)->mixedCase()->numbers()->symbols()` to ensure security standards independent of frontend indicators.

- **Subscription Infrastructure:**
  - `SubscriptionController`: `current()` returns active subscription with plan details.
  - `upgrade(plan_slug)` creates `simulated_payments` record with `status = pending`, creates/updates `subscriptions` with `plan = pending`.
  - Simulated payment processor: a queued job (database driver) waits 3 seconds, then flips `simulated_payments.status` to `success`, updates `subscriptions.plan` to target plan, **does NOT change `users.role`** (no premium role).
  - `cancel()` sets `subscriptions.cancelled_at = now()` but keeps `plan` active until `expires_at`.
- **Feature Gating:**
  - `PlanQuotaService`: central service to check limits. Methods: `canCreateHabit($user)`, `canJoinGroup($user)`, `canUploadPhoto($user, $habitLog)`, `canGeneratePdf($user)`.
  - Returns `['allowed' => bool, 'reason' => string, 'limit' => int, 'used' => int]`.
- **Free Tier Quotas (Hard Limits):**
  - Enforced at controller level before creation. If exceeded, return 422 with `{ error: 'quota_exceeded', resource: 'habits', limit: 5, used: 5 }`.

### PWA Implementation (Phase 1)

  - Create `manifest.json`: `name` ("Momentum Tracker"), `short_name` ("Momentum"), `start_url` ("/"), `display` ("standalone"), `theme_color`, `background_color`, icons (192x192, 512x512).

### Backend Logic

- **Timezone Handling (Architectural Fix):**
  - **NO dynamic modification of `config('app.timezone')`** via middleware to prevent race conditions in concurrent requests.
  - Create a global helper function `user_now($user = null)`: Returns `Carbon::now($user->settings->timezone ?? config('app.timezone'))`.
  - All "today", "yesterday", "isDueToday" calculations use `user_now()`.
  - `habit_logs.logged_date` is always stored in the user's local date (e.g., `user_now()->toDateString()`), NOT UTC.
- **Complex Scheduling & High-Performance Lookups (Habits):**
  - `HabitController`: `store()` validates `schedule` JSON structure based on `frequency`.
    - `daily`: Populates `due_days_of_week` string (e.g., "1,2,3,4,5").
    - `weekly`: Populates `due_days_of_week` string (e.g., "1" for Monday).
    - `custom`: Keeps `schedule` JSON for interval logic.
  - `isDueToday($habit)`: For `daily`/`weekly`, uses high-speed string search/check on `due_days_of_week` against `user_now()->format('N')`. For `custom`, parses JSON interval logic.
- **Automatic Missed Detection (Critical Timezone-Aware Fix):**
  - `MarkMissedHabitsCommand` (Artisan): **Runs every 15 minutes** via Laravel Scheduler to handle global timezone offsets safely.
  - Logic: Fetches active users. Calculates their "local yesterday" using `user_now($user)->subDay()->toDateString()`. Identifies habits due on that date where no `habit_log` exists (or exists with `status = pending`), and updates/creates a log with `status = missed`. The 15-minute interval ensures that even users in `UTC-12` are accurately processed *after* their local midnight has passed, without race conditions.
- **Reminder Dispatcher (Critical Addition):**
  - `DispatchRemindersCommand` (Artisan): Runs every minute via Scheduler.
  - Logic: Query habits where `reminder_time` is not null. Check if current UTC time matches the `reminder_time` adjusted for user's timezone offset. If match, broadcast a `ReminderDue` event via Laravel Reverb (or dispatch to a notification queue).
- **Streak Caching:**
  - `StreakService`: `calculate($habitId, $userId)` — fetches last 30 days of `habit_logs` for habit, computes consecutive `completed` days from `last_log_date` backward.
  - **Exclusions:** Excludes days marked as `skipped` OR days that exist in the `streak_freezes` table for that specific habit.
  - `HabitLogObserver`: on `created`/`updated`/`deleted`, triggers `StreakService::recalculate($habitId)`.
  - All read operations fetch streak from `streaks` table, never compute on the fly.
- **Archiving Logic:**
  - `HabitController`: `destroy()` sets `archived_at = now()`.
  - `restore()` sets `archived_at = null`.
  - `forceDelete()` permanently deletes habit + logs + streak.
  - Global scope `ActiveHabitScope` applies `whereNull('archived_at')` on all `Habit` queries unless explicitly removed.
- **Subscription-Aware Quotas:**
  - `HabitController::store()` calls `PlanQuotaService::canCreateHabit($user)`.
  - Query: `count(habits where user_id = X and archived_at is null) < plans.max_active_habits`.
  - If exceeded: 422 `{ error: 'quota_exceeded', resource: 'habits', limit: 5, used: 5 }`.
- **Categories CRUD & Soft Deletes:**
  - `CategoryController`: full CRUD. `destroy()` executes soft delete (`deleted_at = now()`). 
  - Global scope applied to ensure `whereNull('deleted_at')` is standard for all category queries.
  - `destroy()` checks if category has active habits. If yes: either move habits to `is_default = true` category or return 422 `{ error: 'category_has_habits', count: N }`.
- **Tags with Autocomplete:**
  - `TagController`: `index(q)` searches `tags` where `name LIKE %q%` and `user_id = auth()->id()`.
  - `store()` normalizes name to lowercase, prevents duplicates via unique slug per user.
- **Settings Management:**
  - `SettingsController`: `updateProfile()` (name, avatar, bio), `updatePreferences()` (timezone, theme, language).

### Backend Logic

- **Polymorphic Logs:**
  - `HabitLogController::store()` validates `value` JSON schema based on parent `Habit.type`.
- **Offline Sync Conflict Resolution (Last-Write-Wins Fix):**
  - If request contains `offline_id`, attempt to insert the log.
  - If a `QueryException` occurs due to the `unique_habit_log_per_day` constraint violation (meaning a log for this habit/date already exists, possibly created from another device), catch the exception.
  - Execute an `UPDATE` on the existing record with the incoming payload (overwriting old data with the newest offline sync).
  - Return the updated log with an appended JSON flag: `{ ...log, is_merged: true }` so the frontend can notify the user if needed.
- **Skip with Reason:**
  - `status` = `skipped` requires `skip_reason`.
  - `StreakService`: treats `skipped` as a non-breaking day.
- **Photo Evidence:**
  - `FileController::upload()`. Storage path: `photos/{user_id}/{uuid}.jpg`.
  - **Quota check:** `PlanQuotaService::canUploadPhoto()` counts `habit_photos` for the *specific `habit_log_id`*. Free users limited to 1 photo per individual log entry.
  - Resize to max 1920px.
- **HTML Sanitization:**
  - Uses `mews/purifier`. Restricts to tags: b,i,ul,ol,li,p,br.
- **Timer Habits:**
  - Backend stores `timer_duration`, `started_at`, `ended_at`. Validates time delta (±5s tolerance).
- **Journal Aggregation:**
  - `JournalController::daily($date)` fetches logs + journal entries, merges chronologically. Eager-loads `habit_photos`.

## Cross-Phase Backend Commands (Artisan & Scheduler)

**Scheduler Setup (`app/Console/Kernel.php`):**
```php
protected function schedule(Schedule $schedule)
{
    // 1. Mark pending habits as missed (Runs every 15 mins to safely cover global timezone offsets without race conditions)
    $schedule->command('habits:mark-missed')->everyFifteenMinutes();

    // 2. Dispatch smart/manual reminder notifications
    $schedule->command('reminders:dispatch')->everyMinute();

    // 3. Pre-compute insights, correlations, and tips (Off-peak hours, chunked to prevent OOM)
    $schedule->command('insights:generate')->dailyAt('03:00');

    // 4. Compute leaderboard snapshots
    $schedule->command('leaderboard:compute --period=weekly')->weeklyOn(0, '23:59'); // Sunday
    $schedule->command('leaderboard:compute --period=monthly')->lastDayOfMonth('23:59');

    // 5. Subscription maintenance
    $schedule->command('subscriptions:clean-expired')->daily();

    // 6. PDF quota reset
    $schedule->command('pdf-quota:reset-monthly')->monthlyOn(1, '00:00');
    
    // 7. Clean up old generated PDFs (older than 90 days)
    $schedule->command('reports:clean-old')->weekly();
}
```

**Command Registry:**
- `habits:mark-missed` — Iterates users, calculates their local "yesterday", finds due habits with no log/pending log, and sets them to `missed`. Skips users whose local time hasn't passed midnight yet.
- `reminders:dispatch` — Checks `reminder_time` and `smart_reminders`, fires Reverb broadcasts/notifications.
- `insights:generate` — Chunks active users by ID and dispatches individual queue jobs per user to run `InsightService` and `CorrelationService`.
- `leaderboard:compute` — Aggregates XP/completions, populates `leaderboard_entries` including `tie_breaker_timestamp`.
- `subscriptions:clean-expired` — Sets `status = expired` where `expires_at < now()`.
- `reports:clean-old` — Deletes PDF files from disk where `created_at < 90 days ago`.
- `pdf-quota:reset-monthly` — Sets `users.pdfs_generated_this_month = 0`.

**Configuration Files:**
- **`config/momentum.php`**: Created to store tunable algorithmic constants (e.g., `insights.risk_weights`, `groups.completion_threshold_percentage`) keeping business logic flexible and separate from service classes.
